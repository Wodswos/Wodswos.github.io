    Chrome

    在图标上，Chromium像是没有上色的Chrome（Chrome用的Google代表色：红黄蓝绿）。

    新功能会率先在Chromium推出，而后在Chrome推出。

    这就跟竞技比赛一样，Chromium就相当于所有的参赛成员；Chrome dev就好比预赛出线的选手；接下去是复赛，获胜者将成为Chrome Beta，它们将进入最后的决赛；每个赛季的冠军将是Chrome stable。

    Chromium OS

    Chromium OS is an open-source project that aims to build an operating system that provides a fast, simple, and more secure computing experience for people who spend most of their time on the web.

    Google作为一个后来者身份的互联网公司（1998），在基础软硬件领域自然会被传统的巨头微软（1975）、苹果（1976）等公司疯狂压制。

    但微软在浏览器建设上的傲慢（IE多难用不必多说），加上Google的推波助澜，最终成就了Chromium。而对于Chromium，Google自然不会让他仅仅停留在一个浏览器的阶段，而是希望借助它达到像基础软硬件领域拓展的目的。

    NodeJS

    Node.js 是一个基于 Chrome V8 引擎的 JavaScript 运行环境。 Node.js 使用了一个事件驱动、非阻塞式 I/O 的模型。

    Node 是一个让 JavaScript 运行在服务端的开发平台，它让 JavaScript 成为与PHP、Python、Perl、Ruby 等服务端语言平起平坐的脚本语言。

    发布于2009年5月，由Ryan Dahl开发，实质是对Chrome V8引擎进行了封装。

    Node对一些特殊用例进行优化，提供替代的API，使得V8在非浏览器环境下运行得更好。V8引擎执行JavaScript的速度非常快，性能非常好。Node是一个基于Chrome JavaScript运行时建立的平台， 用于方便地搭建响应速度快、易于扩展的网络应用。Node 使用事件驱动， 非阻塞I/O 模型而得以轻量和高效，非常适合在分布式设备上运行数据密集型的实时应用。

    nvm

    Node Version Manager

    感觉类似于Python中的conda等管理工具，都是管理多个js解释器/环境或者多个python解释器/环境。

    在Node中管理的重点往往是版本，即不同解释器的版本，不同版本的解释器兼容性较差，而一些项目如gitbook、hexo是基于旧版本的（血泪的教训）。

    在Python中管理的重点往往是环境（解释器+库），python解释器小版本之间的兼容性比较强（大版本也逐渐以Python3为主了），反而各种库成为主要的管理对象，如Tensorflow1和Tensorflow2那简直就不像是同源的东西，所以可能会需要两个Python解释器/环境，一个装Tensorflow1的库，另一个装Tensorflow2。

    以上，仅个人看法，毕竟俺也确实不太懂JS

    从长远来看，nvm还是很有必要的，能很大的方便JS的调试之类的工作。

    npm

    官方的意思是， npm 包括三样东西——website，CLI，registry。

    Use the website to discover packages, set up profiles, and manage other aspects of your npm experience. For example, you can set up organizations to manage access to public or private packages.

    The CLI runs from a terminal, and is how most developers interact with npm.

    The registry is a large public database of JavaScript software and the meta-information surrounding it.

    大部分场合下， npm 是随同NodeJS一起安装的包管理工具，能解决NodeJS代码部署上的很多问题，常见的使用场景有以下几种：

    • 允许用户从NPM服务器下载别人编写的第三方包到本地使用。
    • 允许用户从NPM服务器下载并安装别人编写的命令行程序到本地使用。
    • 允许用户将自己编写的包或命令行程序上传到NPM服务器供别人使用。

    命令和选项

    • npm list

    查看相应目录或全局的模块、包

    • -g

    标识全局 global

    若在安装时， -g 指将安装包放在全局目录（ npm 安装目录），Windows（至少我的是）为 C:\Users\Username\AppData\Roaming\npm\node_modules 。能在包括命令行的任何地方调用。

    相应的，若没有 -g 选项，默认为当前目录执行，如果是安装则放在 ./node_modules

    这跟 pip 就不一样， pip 哪有在当前目录安装的道理，必然是依附于特定python解释器的呀

    • npm init

    创建模块，同时会新建一个 package.json 文件

    感觉跟 pip 的职责差异越来越明显了。

    • npm adduser
    • npm publish

    这两个命令上来就是浓浓的 git 气息。所以大致目前的个人理解 npm = pip + git （或者是≈ mvn ？）

    package.json

    package.json 位于模块的目录下，用于定义包的属性。

    • name - 包名。
    • version - 包的版本号。
    • description - 包的描述。
    • homepage - 包的官网 url 。
    • author - 包的作者姓名。
    • contributors - 包的其他贡献者姓名。
    • dependencies - 依赖包列表。如果依赖包没有安装，npm 会自动将依赖包安装在 node_module 目录下。
    • repository - 包代码存放的地方的类型，可以是 git 或 svn，git 可在 Github 上。
    • main - main 字段指定了程序的主入口文件，require(‘moduleName’) 就会加载这个文件。这个字段的默认值是模块根目录下面的 index.js。
    • keywords - 关键字

    关于镜像 

    # 设置镜像
npm config set registry http://registry.npm.taobao.org
# 或单次指定镜像
npm install -g somepackage --registry=https://registry.npm.taobao.org
# 或用定制的cnpm工具
npm install -g cnpm --registry=https://registry.npm.taobao.org
# 或设置环境变量 ELECTRON_MIRROR
set ELECTRON_MIRROR=http://npm.taobao.org/mirrors/electron/

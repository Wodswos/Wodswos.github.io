    C++历史沿革

    C面向机器编程，Cpp面向编译器编程。

    C++ 之父曾经亲口说过，这个世界上并没有一种叫做 C/C++ 的语言。

    我认为，把他们混淆在一起的结果是你两者都学不好。

    很多人只需要用 C，我会规定这些项目只使用 C ，然后实际上他们的代码质量提高了（因为无需混杂 C 跟 C++ 代码）。

    对于必须使用 C++ 的项目而言，要么规定 C++ 局限在某些模块，要么整个项目都使用 C++ 的思路去做。混杂风格更多的出现在国内作坊，看到那些代码就想呕吐。国际上靠谱的码农同行们写的 C++ 大多数都没有这些问题。

    C 和 C++ 值得深入学习吗？ - pansz的回答 - 知乎 https://www.zhihu.com/question/21813639/answer/19397482

    C with class

    1979年，Bjame Sgoustrup到了Bell实验室，开始从事将C改良为带类的C（ C with classes ）的工作。

    第一款C with classes编译器叫Cfront。可以理解为C编译器套壳，先将C with classes的代码翻译为原生C代码再编译。

    The first C++ compiler (Cfront) was written in C++.

    To build that, I first used C to write a “C with Classes”-to-C preprocessor. “C with Classes” was a C dialect that became the immediate ancestor to C++.

    That preprocessor translated “C with Classes” constructs (such as classes and constructors) into C. It was a traditional preprocessor that didn’t understand all of the language, left most of the type checking for the C compiler to do, and translated individual constructs without complete knowledge. I then wrote the first version of Cfront in “C with Classes”.

    虚函数、函数重载、引用机制、const关键字、双斜线单行注释等特性被加入C++。

    1983年，该语言被正式命名为C++。

    1985年，Stroustrup的C++参考手册《C++ Programming Language》出版，并成为当时的 C++ 实践标准。

    1990年，《Annotated C++ Reference Manual》发布，同年，Borland公司的商业版Turbo C++编译器问世。Turbo C++附带了大量函数库，该编译器至今仍有使用。

    1993年，由于很难整合C++的异常机制，Cfront退出了历史的舞台。

    C++标准化

    Standard Template Library 由 Alexander Stepanov 创造于1979年前后，和C++孕育的年代相似。

    STL的代码从广义上讲分为三类：algorithm（算法）、container（容器）和iterator（迭代器），几乎所有的代码都采用了模板类和模板函数的方式，这相比于传统的由函数和类组成的库来说提供了更好的代码重用机会。

    在C++标准中，STL被组织为以下13个头文件： <algorithm>、<deque>、<functional>、<iterator>、<vector>、<list>、<map>、<memory>、<numeric>、<queue>、<set>、<stack>、<utility>

    其中，vector 的底层为顺序表（数组），list 的底层为双向链表，deque 的底层为循环队列，set 的底层为红黑树，hash_set 的底层为哈希表。

    在通过了第一个草案之后，委员会投票并通过了将STL包含到C++标准中的提议。STL对C++的扩展超出了C++的最初定义范围，也因此延缓了C++标准化的进程。

    1998年，C++标准委员会发布了C++语言的第一个国际标准— ISO/IEC 14882:1998 ，该标准即为大名鼎鼎的C++98。

    2005年，C++标准委员会发布了一份技术报告（Technical Report，TR1）详细说明了计划引入C++的新特性。

    C++ 2.0

    2011年中，新的 C++标准 （C++11）面世。

    • 正则表达式（ 正则表达式详情 ）
    • 完备的随机数生成函数库
    • 新的时间相关函数
    • 原子操作支持
    • 标准线程库（2011之前，C和C++语言均缺少对线程的支持）
    • 一种能够和某些语言中foreach语句达到相同效果的 新的for语法
    • auto关键字
    • 新的容器类
    • 更好的union支持
    • 数组初始化列表的支持以及变参模板的支持

    Boost 库对该版本影响很大，一些新的模块甚至直接衍生于Boost中相应的模块。

    新式头文件不带副名称 .h ，例如 #include<csdio> ，但旧式仍可用，如 #include<stdio.h> ，但不推荐

    2014年8月18日，经过C++标准委员投票，C++14标准获得一致通过。

    2017年12月5日，ISO C++ 委员会正式发布了 C++ 17 标准。

    LLVM

    LLVM 是构架编译器的框架系统，以 C++ 编写而成。用于优化以任意程序语言编写的程序的编译时间(compile-time)、链接时间(link-time)、运行时间(run-time)以及空闲时间(idle-time)。

    LLVM 命名最早源自于底层虚拟机（ L ow L evel V irtual M achine）的缩写,由于命名带来的混乱，LLVM就是该项目的全称。

    Windows

    Windows环境下使用GNU

    一般来说都是用编辑器如 Visual Studio Code 配合 GNU 下 C++的开发流程。

    Cygwin

    原Cygnus出品，被RedHat收购，提供运行于Windows平台的类Unix环境。

    让Windows用户不仅可以使用GNU工具，理论上Linux下的代码可以通过Cygwin重新编译移植到Windows（将部分POSIX调用转换成Windows的API调用）。

    如果程序只用到C/C++ 标准库，可以用MinGW 或Cygwin 编译。

    如果程序还用到了 POSIX API ，则只能用Cygwin 编译，Cygwin会尽量转换为对应的Win32API。但Cygwin编译运行，需要依赖cygwin1.dll，而 MinGW 编译出的程序不需要第三方DLL。

    MinGW

    Minimalist GNU for Windows.

    提供给Windows用户GNU工具包，使得可以在Windows下使用GCC等。

    https://github.com/FrankHB/pl-docs/blob/master/zh-CN/mingw-vs-mingw-v64.md

    MinGW 是 GNU 工具（包括编译器 GCC 和 GNU binutils 和调试器 GDB 等）在 Win32 上的一个移植，是从 Cygwin 里 fork 出来的。当初只考虑 32 位。和 Cygwin 相比，不强调 POSIX 兼容性而相对强调性能和减小依赖。

    然后，因为只支持 32 位，有人觉得不够用。这里的一个主要人物，就是现在 MinGW-W64 的主要维护者 Kai Tietz 。 因为工作需要重新实现提供 x64 支持的 MinGW 后提交到上游但 被拒绝 ，于是 fork 为单独的项目，这就是 MinGW-W64 的由来。

    因此MinGW和MinGW-w64只是名字像，它们是两个不同的项目。

    MinGW 只能编译生成32位可执行程序，而 MinGW-w64 则可以编译生成64位或32位可执行程序。

    MinGW发行版，如MSYS2、TDM-GCC等，类似于 Linux 内核和 Linux 发行版的关系。
